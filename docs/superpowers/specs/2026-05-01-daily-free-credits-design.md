# 每日免费额度设计方案

## 背景与目标

当前系统处于邀测阶段，用户通过邀请码注册，并获得 `invites.initial_credits` 指定的一次性初始额度。生成任务创建时扣 1 点，失败后退 1 点，管理员可以手动调整用户余额。

后续项目会商业化，用户将通过付费获得额度。因此“每天给用户一个免费体验额度”的能力不能直接重置现有总余额，否则会和未来付费额度混在一起，带来资损、对账和运营规则问题。

本次目标是建立可商业化演进的额度模型：

- 用户每天拥有固定免费体验额度。
- 每日刷新只恢复免费额度，不影响未来付费额度。
- 生成时优先消耗免费额度，免费额度不足时再消耗付费额度。
- 注册邀请码继续决定用户的初始免费体验额度上限。
- 账本能区分免费额度刷新、免费额度消耗、付费额度消耗和退款来源。

本次不实现支付、订单、套餐、发票、优惠券、订阅或复杂财务报表。

## 推荐方案

采用“双钱包”设计：免费日额度钱包和付费额度钱包分离。

用户表保存：

- `daily_free_credit_limit`：每日免费额度上限。
- `daily_free_credit_balance`：当天剩余免费额度。
- `paid_credit_balance`：付费额度余额。
- `last_daily_free_credit_refreshed_on`：上次刷新免费额度的业务日期。

注册时，邀请码的 `initial_credits` 不再只是一次性余额，而是该用户的每日免费额度上限。注册完成后：

- `daily_free_credit_limit = initial_credits`
- `daily_free_credit_balance = initial_credits`
- `paid_credit_balance = 0`
- `last_daily_free_credit_refreshed_on = 当前业务日期`

旧字段 `credit_balance` 在迁移期可以保留为兼容字段，但新的业务逻辑以双钱包字段为准。前端短期仍可展示一个合计额度：`daily_free_credit_balance + paid_credit_balance`，等后续商业化页面准备好后再拆开展示。

## 每日刷新规则

每日刷新只作用于免费钱包：

- 如果用户是活跃状态，并且 `last_daily_free_credit_refreshed_on` 早于当前业务日期，则把 `daily_free_credit_balance` 设置为 `daily_free_credit_limit`。
- 不修改 `paid_credit_balance`。
- 不累计未用完的免费额度。
- 免费额度上限为 0 的用户仍更新刷新日期，但余额保持 0。
- 管理员禁用用户是否刷新不影响资金安全；推荐只刷新 active 用户，禁用用户重新启用后在下一次请求中惰性刷新。

业务日期默认使用数据库时区下的 `CURRENT_DATE`。生产环境应让数据库时区与产品运营时区保持一致；如果后续需要多时区用户策略，再单独引入配置。

## 刷新触发机制

采用“惰性刷新 + 后台定时刷新”组合。

惰性刷新是强一致入口：用户登录、查询当前用户、创建生成任务前，系统先检查并刷新该用户的免费额度。这样即使后台任务没有运行，用户当天第一次使用时也能拿到免费额度。

后台定时刷新是体验和运营兜底：服务进程启动一个每日任务，在运营日切换后批量刷新所有 active 用户。该任务让后台用户列表和用户余额更早反映当天额度，但不能作为唯一正确性来源。

为了避免多进程部署重复执行造成账本重复，刷新逻辑需要幂等：

- 用户行更新时带上 `last_daily_free_credit_refreshed_on < 当前业务日期` 条件。
- 只有实际更新了余额的用户才写刷新账本。
- 刷新账本可以用 `user_id + ledger_type + business_date` 建唯一约束，防止并发重复记录。

## 消耗与退款规则

生成任务仍然创建成功后扣 1 点。

扣费顺序：

1. 如果 `daily_free_credit_balance >= 1`，扣免费额度。
2. 否则如果 `paid_credit_balance >= 1`，扣付费额度。
3. 两个钱包都不足时返回额度不足。

退款必须退回原扣费钱包。生成扣费账本需要记录本次任务扣的是免费额度还是付费额度。任务失败退款时根据该任务的扣费账本恢复对应钱包，保持现有幂等退款语义，避免重复退款。

管理员手动“加点/扣点”调整付费额度，并在文案中称为“付费/通用额度调整”。每日免费额度上限由邀请码和未来后台配置单独管理，不通过手动余额调整入口修改。

## 账本设计

账本需要能表达钱包来源。推荐在 `credit_ledger` 中新增字段：

- `wallet_type`：`daily_free` 或 `paid`。
- `business_date`：免费刷新对应的业务日期，其他类型可为空。

同时扩展账本类型：

- `daily_free_refresh`：每日免费额度刷新。
- `daily_free_generation_debit`：生成任务消耗免费额度。
- `daily_free_generation_refund`：失败任务退回免费额度。
- `paid_generation_debit`：生成任务消耗付费额度。
- `paid_generation_refund`：失败任务退回付费额度。
- `paid_admin_adjustment`：管理员调整付费额度。

现有 `generation_debit`、`generation_refund`、`admin_adjustment` 类型保留用于历史记录。新逻辑写入新的明确类型，避免仅靠 `wallet_type` 推断业务语义。

## 数据迁移

新增迁移应完成：

- 在 `users` 表添加双钱包字段。
- 将现有 `credit_balance` 迁移到 `daily_free_credit_balance`。
- 将 `daily_free_credit_limit` 初始化为当前 `credit_balance`。
- 将 `paid_credit_balance` 初始化为 0。
- 将 `last_daily_free_credit_refreshed_on` 初始化为 `CURRENT_DATE`。
- 扩展 `credit_ledger` 的类型约束和钱包字段。

这意味着当前邀测用户的现有余额会成为他们的每日免费额度上限和当天免费余额。管理员后续新建邀请码时，`initial_credits` 就代表该用户每天可恢复的免费体验额度。

`credit_balance` 可在第一阶段保留并由代码同步为合计值，降低前端和接口改动。等商业化付费页面上线时，再迁移 API 返回结构并移除或废弃该字段。

## API 与前端影响

用户登录和会话返回可以短期保持 `credit_balance` 字段，值为免费余额与付费余额之和，避免当前前端大改。

建议同时增加可选字段：

- `daily_free_credit_balance`
- `daily_free_credit_limit`
- `paid_credit_balance`

创作台显示：

> 今日免费额度 X/Y，付费额度 Z。

生成按钮可继续只判断合计额度是否充足。后端仍是最终约束来源。

管理员后台用户列表新增免费额度和付费额度列，同时保留合计显示，方便运营查看当前可用额度。

## 错误处理

每日刷新失败时：

- 用户登录和生成任务应返回服务器错误，避免在额度状态不确定时继续扣费。
- 后台定时刷新失败只记录日志，下一次惰性刷新仍可修复单个用户。

扣费与任务创建必须在同一个事务内完成。退款与任务失败状态更新也必须在同一个事务内完成。

并发创建任务时仍依靠事务更新用户行和现有“一人一个活跃任务”唯一索引保证一致性。

## 测试策略

后端按 TDD 增加测试：

- 注册用户时，邀请码初始额度会设置每日免费额度上限和当天免费余额。
- 当业务日期变更时，刷新会把免费余额恢复到上限。
- 刷新不会修改付费余额。
- 同一天重复刷新不会重复写账本。
- 生成任务优先扣免费额度。
- 免费额度不足时扣付费额度。
- 两个钱包都不足时创建任务失败。
- 任务失败退款会退回原扣费钱包。
- 管理员调整默认影响付费额度，并写付费调整账本。

前端测试按实现范围补充：

- 创作台展示今日免费额度和付费额度。
- 额度不足时仍显示现有阻断提示。
- 管理员后台能看到拆分额度，或在暂不拆分时继续正确展示合计额度。

## 非目标

- 不接入支付渠道。
- 不设计订单、套餐、订阅或退款流水。
- 不做免费额度按用户等级动态变化。
- 不做免费额度跨天累计。
- 不做多时区用户级刷新。
- 不做复杂财务报表。

## 验收标准

- 每个用户每天都能恢复到自己的免费额度上限。
- 每日刷新不影响付费额度。
- 生成任务优先消耗免费额度，再消耗付费额度。
- 失败退款回到原扣费钱包。
- 刷新账本幂等，不因重复任务或并发导致重复记录。
- 现有邀测用户能通过迁移获得与当前余额相同的每日免费额度上限。
- Go 测试和前端相关测试通过。
